# テスト仕様書 - 営業日報システム

## 1. テスト概要

### 1.1 テストの目的
営業日報システムが要件定義通りに動作し、品質基準を満たすことを確認する

### 1.2 テストの種類

| テスト種類 | 目的 | 実施者 |
|-----------|------|--------|
| 単体テスト（Unit Test） | 個別の関数・メソッドの動作確認 | 開発者 |
| 結合テスト（Integration Test） | API・DB連携の動作確認 | 開発者 |
| E2Eテスト（End-to-End Test） | 画面操作からDBまでの一連の流れ確認 | QA/開発者 |
| 機能テスト（Functional Test） | 機能要件の充足確認 | QA |
| セキュリティテスト | 脆弱性の確認 | QA/セキュリティ担当 |
| パフォーマンステスト | 性能要件の確認 | QA |
| ユーザー受け入れテスト（UAT） | 実ユーザーによる受け入れ確認 | エンドユーザー |

### 1.3 テスト環境

| 環境 | 用途 | URL |
|------|------|-----|
| 開発環境 | 開発中の動作確認 | http://dev.example.com |
| テスト環境 | QAテスト実施 | http://test.example.com |
| ステージング環境 | 本番同等環境でのテスト | https://staging.example.com |
| 本番環境 | 実運用環境 | https://app.example.com |

## 6. セキュリティテスト仕様

### 6.1 認証・認可テスト

| テストケースID | テスト項目 | テスト内容 | 合格基準 |
|---------------|-----------|-----------|----------|
| ST-SEC-001 | SQLインジェクション | 入力欄に `' OR '1'='1` を入力 | SQLインジェクションが防がれる |
| ST-SEC-002 | XSS攻撃 | 入力欄に `<script>alert('XSS')</script>` | スクリプトが実行されない |
| ST-SEC-003 | CSRF攻撃 | CSRFトークンなしでPOST | リクエストが拒否される |
| ST-SEC-004 | パスワード強度 | 7文字以下のパスワード登録 | 登録拒否 |

## 11. テスト完了基準

### 11.1 テスト合格基準

- 単体テスト: カバレッジ80%以上
- 結合テスト: 全テストケース合格
- E2Eテスト: 全主要シナリオ合格
- 機能テスト: 全機能要件を満たす
- セキュリティテスト: 重大な脆弱性なし
- パフォーマンステスト: 全項目が基準値以内
- UAT: エンドユーザーの承認取得

### 11.2 不具合の優先度

| 優先度 | 定義 | 対応 |
|--------|------|------|
| Critical | システムが動作しない、データ損失の可能性 | 即時修正必須 |
| High | 主要機能が使えない | リリース前に修正必須 |
| Medium | 機能に制限があるが回避策あり | リリース前に修正推奨 |
| Low | 見た目の問題、マイナーな不具合 | 次回リリースで修正 |

- GitHub Actions / GitLab CI でテスト自動実行

- カバレッジレポート自動生成
- テスト失敗時は自動通知
